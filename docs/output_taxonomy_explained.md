# What's in the Output, and Who Made It

**Audience:** Anyone trying to understand the files in `output/` without reading code.
**Scope:** Explains the taxonomy of the *current* extraction output and which parts the
LLM agents actually produced.
**Companion doc:** For the full, deep field-by-field reference see
[`docs/data_dictionary.md`](./data_dictionary.md). This page is the short version.

> **One thing to get straight up front.** The data in `output/` is the **catalog
> layer** — a lightweight, searchable index of every law. It is *not* the deep,
> clause-by-clause extraction the LLM agents produce. Those two things are easy to
> confuse, so this doc draws the line clearly.

---

## 1. The two data layers (and why they look different)

| | **Catalog layer** | **Extraction layer** |
|---|---|---|
| **What it is** | One row per law: identity, status, source, and search tags | Many structured facts per law: obligations, penalties, deadlines, rights, thresholds |
| **Where it lives** | `output/laws.json` (and the `*.md`/`*.json` companions in `output/`) | Postgres database (`law_enforcement_details`, `law_triggering_thresholds`, etc.) |
| **How it's made** | Source CSVs + dimension tables + **keyword classification** | The **LLM agents** reading full bill text |
| **In this repo's `output/`?** | ✅ Yes | ❌ No — not exported here |

**Key takeaway:** the taxonomy you see in `output/laws.json` was **not generated by the
LLM agents.** `output/law_tag_categories.json` says so plainly in its own notes:

> *"Derived from summary.csv, dimension/mapping tables, and keyword classification."*

The tell-tale sign is the `ai_topic` vocabulary, which contains keyword-extraction noise
like `ai-healthcare-the`, `frontier`, `user`, and `automated` — tokens scraped from
titles, not concepts an LLM curated. The agents' real work (Section 3) lands in the
database, and `output/extraction_runs/` is currently **empty** (`.gitkeep` only).

---

## 2. The catalog taxonomy (what's actually in `output/laws.json`)

`output/laws.json` holds **244 law records**. Each record carries flat `tags` plus a
`tag_categories` object that groups those tags into **nine controlled vocabularies**:

| Category | What it answers | # of laws tagged | How it's assigned |
|---|---|---|---|
| `jurisdiction` | Which state/body | 244 / 244 | From source metadata (CSV) |
| `state_abbrev` | Two-letter state code | 244 / 244 | From source metadata (CSV) |
| `source` | Where we got the law | 244 / 244 | From source metadata — `Orrick` (188) or `IAPP` (56) |
| `lifecycle_status` | Proposed → passed → in effect | 244 / 244 | Normalized from source status |
| `ai_topic` | Rough subject of the law | 188 / 244 | **Keyword classification** from titles (noisy) |
| `concept` | Curated themes (children, employment, privacy-data, …) | 177 / 244 | Keyword classification against a curated list |
| `ai_scope` | Kind of AI regulated (ADM, generative, frontier…) | 55 / 244 | Partial enrichment |
| `compliance_requirement` | Coarse duty type (assessments, notice…) | 56 / 244 | Partial enrichment |
| `regulated_actor` | developer / deployer / provider | 56 / 244 | Partial enrichment |

The full controlled vocabularies are listed in
[`output/law_tag_categories.md`](../output/law_tag_categories.md). The per-record field
definitions are in [`output/laws_fields.md`](../output/laws_fields.md).

**Coverage is uneven on purpose.** The first four categories are present on every law
(they come straight from metadata). The richer categories (`ai_scope`,
`compliance_requirement`, `regulated_actor`) are only filled on ~55–56 laws — the
catalog layer is a search index, not a completeness guarantee.

### Snapshot of the 244 records
- **By status:** 179 in effect · 16 passed-not-yet-in-effect · 49 proposed
- **By source:** 188 Orrick · 56 IAPP
- **Supporting files:** 219 extracted full-text files in `output/law_texts/`,
  22 set aside in `output/law_texts_quarantine/`, 236 raw sources in `output/law_sources/`.

> Note the count mismatch: `laws.json` has 244 rows, the authoritative law list
> (`data/fact_laws.csv`) is ~232, and there are 219 text files. These are different
> snapshots taken at different times — treat `data/fact_laws.csv` as the source of
> truth for "how many laws," and `laws.json` as a catalog export that may run ahead or
> behind.

## 3. The LLM agents (what they actually do)

The agents are the deep-extraction engine. They read the **full statutory text** and
emit structured legal facts into the database — the **extraction layer**, *not* the
catalog tags above. All run on **local models via LM Studio** (`openai/gpt-oss-20b`);
there is no Anthropic API in this project.

There are two batteries plus a gatekeeper.

### A gate runs first
- **`section_triage`** — decides, passage by passage, whether text is AI-relevant enough
  to send to the extraction battery. Keyword pre-screen first (free), LLM only on the
  fence-sitters. Keeps the expensive agents off boilerplate.

### Clause-level battery — 6 agents, read the bill *passage by passage*
| Agent | Extracts | Plain English |
|---|---|---|
| **`obligation`** | obligations + their timelines + enforcement, in one pass | "What must someone *do*?" (the `shall`/`must`/`prohibited` clauses) |
| **`definition_actor`** | definitions, actor-role mappings, framework references | "What do the words mean, and who's who?" |
| **`threshold_exception`** | thresholds + exceptions | "When does a duty kick in, and when is it waived?" |
| **`rights_protection`** | individual rights | "What are people *entitled to*?" (notice, opt-out, appeal, deletion) |
| **`compliance_mechanism`** | procedural duties | "How do you prove compliance?" (impact assessments, audits, recordkeeping) |
| **`preemption`** | jurisdictional conflicts | "Does this clash with federal law or other states?" |

### Bill-level battery — 3 agents, read the *whole bill at once*, once per law
| Agent | Populates | Plain English |
|---|---|---|
| **`applicability_agent`** | covered entities/sectors, AI types in scope, size thresholds, exemptions | "Does this law apply to me?" |
| **`enforcement_agent`** | enforcing body, max penalty, penalty unit, cure period, private right of action | "What happens if I don't comply?" |
| **`compliance_timeline_agent`** | deadlines, effective dates, frequencies, sequencing | "By when?" |

Every fact an agent emits is tied to a **verbatim quote** ("evidence span") from the law
and graded for confidence (an A–D trust tier). The field-level detail of what each agent
writes is documented in [`docs/data_dictionary.md`](./data_dictionary.md).

> The `obligation`, `definition_actor`, and `threshold_exception` agents are
> *consolidated* — each does the job of several older agents in one pass, because those
> facts sit next to each other in legislative text. That's why the code comments mention
> a "7-agent" registry while the pipeline behaves as 6 clause-level + 3 bill-level.

---

## 4. How to read this in one sentence

> `output/laws.json` is a **keyword-built catalog** of 244 AI laws you can filter and
> search; the **nine LLM agents** build a far richer, quote-grounded extraction in the
> database that this `output/` folder does not currently export.

## 5. Open questions worth confirming

1. **Do agent outputs ever feed back into the catalog tags?** Today the catalog appears
   fully rule/keyword-derived. If the plan is to replace the noisy `ai_topic` keywords
   with agent-derived concepts, that hand-off isn't visible in `output/` yet.
2. **`output/extraction_runs/` is empty.** If a run is meant to drop artifacts here, the
   exporter for that doesn't appear wired up — worth confirming before relying on it.
3. **Record-count drift** between `laws.json` (244), `fact_laws.csv` (~232), and the text
   files (219) suggests these were generated at different times.
